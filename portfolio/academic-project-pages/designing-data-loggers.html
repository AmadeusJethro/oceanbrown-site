Project 03

Designing Reliable Data Loggers for Event Markets

A software-systems writeup on what I learned by iterating from single-venue collectors to a unified, multi-service event-market observatory deployed on simple but durable cloud infrastructure.

Primary Theme Long-running data systems
System Shape Discovery + polling + buffering + health + persistence
Portfolio Use Software engineering / platform architecture case study

Structured Abstract

Question. What makes a market data logger trustworthy enough for research and operations, rather than just functional enough to scrape an endpoint once?

Approach. I built multiple venue-specific loggers, then consolidated the lessons into a unified collector with structured logging, health monitoring, buffered writes, and explicit operational controls.

Main finding. The hard part of a logger is not HTTP; it is lifecycle management: rate limits, stale-feed detection, recovery behavior, schema discipline, cloud cost, and data provenance.

Why it matters. Those same concerns dominate energy data systems, ISO ingest pipelines, and long-running quantitative telemetry infrastructure.

System Evolution

From venue-specific collectors to a unified observatory

The early loggers were venue-specific: one for Polymarket and one for Kalshi, each with its own API client, database connection, deployment script, and health logic. That was useful because each venue had different rate limits, schemas, and endpoint behavior. But the real systems lesson emerged later: once the research question became cross-venue, the architecture had to become modular enough to normalize both feeds while still respecting their differences.

The more mature unified collector pulled those lessons together into explicit services: discovery, flat-rate polling, trade collection, arbitrage detection, buffered database writes, and health monitoring. That is the point where the work stopped being a collection of scrapers and became a real data platform.

Architecture Lessons

What ended up mattering most

Concern Practical lesson
Discovery Fetch the market universe separately from polling so startup is fast and market lists can be refreshed safely.
Polling Rate-awareness and predictable cadence matter more than ad hoc scraping speed.
Writes Batching via a tick buffer reduces database overhead and makes throughput manageable.
Health A logger that cannot detect stale feeds, dropped ticks, or silent partial failure is not operationally trustworthy.
Schema Cross-venue analysis becomes possible only once source-specific quirks are normalized into a common storage model.
Software takeaway The highest-value design decision was separation of concerns. Discovery, collection, buffering, health, and analytics should not be entangled if the system is expected to run unattended for long periods.

Operational Lessons from the Unified Collector

What the mature system included

  • structured logging with machine-readable health and status events
  • per-venue clients so one platform's problems did not poison the entire process
  • a tick buffer to batch writes instead of hammering the database
  • background discovery refreshes so data collection could start quickly and improve coverage later
  • health checks for stale data, missing ticks, and service degradation
  • graceful shutdown and explicit lifecycle management for long-running collectors

That set of features sounds mundane until a collector has to survive real runtime conditions. At that point, heartbeat visibility and restart safety are more valuable than squeezing out another small throughput win.

Cloud Cost and Deployment Perspective

Why simple GCP choices were often correct early on

The venue-specific collectors were deployed with deliberately simple infrastructure: Cloud Build, Cloud Run, and Cloud SQL. One Cloud Run configuration used 512Mi memory, 1 CPU, a single always-on instance, long timeouts, and Cloud SQL attachment. That is not glamorous, but it is often the right first production shape for an always-on collector: low operational burden, fast redeploys, and a clean path from local Docker to hosted service.

Cost lesson Cost is architecture. A logger that is mathematically elegant but operationally expensive is not a good logger. In early-stage systems, the right question is often: what is the simplest hosted shape that preserves reliability without creating a full-time ops problem?

The main tradeoff is also clear: serverless simplicity is excellent until the workload becomes too stateful, too latency-sensitive, or too constant for that model. Knowing where that boundary sits is part of the design work.

Security and Data Hygiene

Public data still needs discipline

A recurring systems insight was that public data collection does not remove the need for security posture. The better repos used explicit HTTP requests, kept private keys out of scope, hashed raw responses for auditability, and treated third-party datasets as untrusted input rather than something to execute or trust blindly.

  • keep credentials out of notebooks and committed files
  • store raw datasets under separate data roots and keep them out of Git
  • hash externally acquired datasets and snapshots
  • separate research environments from sensitive local profiles and cloud credentials

Why This Belongs on a Portfolio

It is more than scraper engineering

I would frame this project as a study in trustworthy telemetry. The work shows how data systems mature: from endpoint wrappers to observability-aware platforms that can support research, monitoring, and eventually trading. That is useful well beyond event markets. The same design pressures appear in ISO ingestion pipelines, operational dashboards, market surveillance, and any quantitative workflow that depends on long-running external feeds.
