Project 04

Evaluating Agentic Systems for Quant Workflows

A project on orchestration, guardrails, and benchmark design for multi-agent research systems intended to support quantitative and time-series workflows without silently introducing bad methodology.

Primary Theme Agent orchestration and evaluation
System Role-specialized multi-agent pipeline with evidence-backed scoring
Portfolio Use Agentic systems / quant workflow infrastructure

Structured Abstract

Question. Can a specialized multi-agent system reduce methodological errors in quantitative workflows without damaging output quality?

Approach. I built an evaluation harness for a custom agent pipeline, including routing checks, compliance cases, artifact-backed manifests, and paired benchmark infrastructure for guarded versus direct-model runs.

Main finding. The crucial problem was not only whether agents produced plausible output, but whether the system could prove that the right workflow actually happened. That pushed the project toward artifact-first evaluation and release-gate governance.

Why it matters. Quant teams cannot safely rely on agentic systems unless routing, review, and methodological constraints are both enforced and auditable.

System Thesis

Useful-looking output is not enough

The underlying orchestration idea was strong: separate roles for context gathering, research, planning, implementation, security review, optimization, quantitative review, and audit. In principle, a role-specialized agent system should outperform a monolithic assistant on safety-critical quantitative tasks because the pipeline encodes deliberate review points.

But the core research lesson was subtler. A multi-agent system can produce good-looking work without proving that the intended path ran. If the benchmark only reads text, it may over-credit a system that skipped the very review stages meant to make it safe.

What I Built

Evidence-first benchmark infrastructure

  • case fixtures for compliance and routing scenarios
  • an execution runner that records raw event streams and preserved artifacts
  • canonical run manifests with hash chains
  • derived pipeline-completion evidence artifacts
  • scoring logic with routing, compliance, output, and execution-failure dimensions
  • grounded dataset governance with train-dev, public-test, and private-holdout splits
  • release-gate logic for human quality evidence, routing evidence, reproducibility, and holdout use
Design principle The harness moved from text-first evaluation to artifact-first evaluation. If the system cannot emit a canonical completion artifact, it should not receive full credit for routing success.

Main Findings

1. Routing quality must be measured through evidence, not prose

The benchmark initially failed to score routing correctly because subagent execution did not always appear in the parent session as explicit spawn events. The fix was architectural: manifests, pipeline evidence artifacts, and preserved run directories became the trusted signal, not whatever the final answer happened to say.

2. Guardrails need their own governance

Once the scorer became more sophisticated, it introduced a new class of risk: the system could appear to improve because the measurement changed, not because the underlying agent behavior improved. That led to shadow-scoring, reason codes, release gates, and explicit checks for dataset provenance and human-quality evidence.

3. Quantitative workflows need domain-specific failure tests

The most useful benchmark cases were not generic prompt tests. They were time-series and backtest failure modes: KFold on temporal data, shuffle leakage, negative shifts, missing Citadel-style backtest disclosures, and routing failures where the system skipped specialized review stages. 

Release-gate blockers captured by the harness
- missing human-quality evidence
- reproducibility runs missing
- final holdout not used
- routing completion evidence missing
The release gate is deliberately conservative: benchmark readiness is blocked until both methodology and evidence are complete.

Why This Is Interesting for Quant Teams

Agentic workflows should look like audited research processes

The interesting portfolio angle is not "I built agents." It is that I treated agentic systems as workflow infrastructure for quantitative research. That means the system has to do more than answer questions: it has to route work to the right reviewer, preserve evidence, expose failure modes, and refuse to claim completion when the benchmark basis is weak.

  • For quant teams: safer automation for research, backtesting, and code review.
  • For software teams: better orchestration, benchmark design, and release governance.
  • For regulated or high-stakes workflows: explicit proof of what ran and what passed.

How I Would Frame It Publicly

Portfolio thesis

I would present this as a project on agentic research infrastructure: how to make specialized agent systems useful for quantitative workflows by attaching them to evidence, evaluation, and methodological review. The strongest statement is not that the agents are magical. It is that the benchmark now knows when not to trust them.
